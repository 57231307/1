## 一、已完成的整改工作

### 1.1 响应格式统一

**问题**：前后端响应格式不统一，部分模块手动解析响应
**解决方案**：
- 统一使用 `ApiResponse<T>` 包装所有 API 响应
- 前端 `ApiService` 自动处理响应解析
- 后端所有接口返回 `ApiResponse` 格式

### 1.2 API 路径统一

**问题**：API 路径前缀重复（`/api/v1/erp` 同时出现在 ApiService 和各个 service 文件中）
**解决方案**：
- 移除 service 文件中的重复路径前缀
- 统一由 `ApiService` 管理 API 基础路径

### 1.3 数据模型位置统一

**问题**：数据模型定义分散，部分在 service 文件内，部分在 models 目录
**解决方案**：
- 创建 `models/fabric_order.rs` - 面料订单模型
- 创建 `models/purchase_order.rs` - 采购订单模型
- 更新 `models/mod.rs` 导出新模块
- 将模型从 service 文件移到 models 目录

### 1.4 数值类型统一

**问题**：部分模块使用 `f64` 类型，存在精度损失风险
**解决方案**：
- 将面料订单相关的数值字段从 `f64` 改为 `String`
- 包括：数量、价格、重量、宽度等字段

### 1.5 具体模块修复

#### 面料订单模块
- ✅ 重写 `fabric_order_service.rs`，使用 ApiService
- ✅ 修复 f64 类型为 String
- ✅ 移动模型到 `models/fabric_order.rs`
- ✅ 更新 `fabric_order.rs` 页面导入路径

#### 采购订单模块
- ✅ 重写 `purchase_order_service.rs`，使用 ApiService
- ✅ 移动模型到 `models/purchase_order.rs`
- ✅ 修复后端 `purchase_order_handler.rs` 的 delete 方法返回 ApiResponse
- ✅ 修复后端 `purchase_order_handler.rs` 的 delete_item 方法返回 ApiResponse
- ✅ 更新 `purchase_order.rs` 页面导入路径
