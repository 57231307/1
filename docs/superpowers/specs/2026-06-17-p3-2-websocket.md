## 一、目标与背景

### 1.1 业务目标

当前 ERP 通知模块采用**轮询**（前端每 30 秒 GET 一次 `/api/v1/erp/notifications`）实现：
- 延迟高（30 秒轮询周期）
- 浪费带宽（绝大多数请求无新数据）
- 移动端体验差（需要长连接 / 锁屏即断）

升级为**WebSocket 实时推送**后：
- **延迟降低到 < 1 秒**：服务端主动推送
- **节省带宽**：按需推送，无轮询
- **移动端友好**：配合 Push Notification 实现锁屏推送
- **多端同步**：浏览器、平板、手机实时同步

### 1.2 技术目标

- 后端：基于 `axum` + `tokio-tungstenite` 实现 `/api/v1/erp/ws/notifications` 端点
- 前端：封装 `WebSocket` 工具类，支持**自动重连** + **心跳** + **消息分发**
- 多租户隔离：每个 WebSocket 连接绑定 `tenant_id`
- 鉴权：WebSocket 握手时携带 JWT token
- 消息推送：服务端在通知创建时主动推送给该用户的连接

### 1.3 范围

**包含**：
1. 完整 WebSocket 实时通信设计 spec（本文件）
2. 完整实施 plan（`docs/superpowers/plans/2026-06-17-p3-2-websocket.md`）
3. 关键路径 demo：通知模块 WebSocket 实现
   - 后端：`backend/src/websocket/notifications.rs`（WebSocket handler + 连接管理）
   - 路由：`/api/v1/erp/ws/notifications`
   - 前端：`frontend/src/utils/websocket.ts`（封装 + 自动重连 + 心跳）
   - 1 个集成测试（连接 + 推送 + 断开重连）
   - 用户手册
4. 不破坏现有 P0/P1/P2/P3-1 已合入的功能
5. 主项目 `backend/` 复用现有通知 service 发送逻辑

**不包含**（P4+ 后续阶段）：
- 全双工（前后端双向通信）：P3-2 仅做服务端推送
- 集群 WebSocket 网关（需 Redis Pub/Sub 或 NATS）
- 移动端原生 WebSocket（留待 P3-3）
- SSE（Server-Sent Events）替代方案

## 二、决策记录（Q1-Q7 + 矛盾解决）

### 2.1 7 个澄清问题

| 编号 | 问题 | 决策 |
|------|------|------|
| Q1 | 通信协议 | WebSocket（RFC 6455） |
| Q2 | 后端框架 | axum + tokio-tungstenite（与现有 P0/P1/P2 一致） |
| Q3 | 前端封装 | TypeScript class + EventTarget（标准化事件） |
| Q4 | 重连策略 | 指数退避（1s → 2s → 4s → 8s → 16s → 30s 上限） |
| Q5 | 心跳机制 | 客户端每 30 秒发送 ping，服务端回 pong |
| Q6 | 多租户隔离 | WebSocket 握手时验证 JWT 中的 tenant_id |
| Q7 | 是否合到 main | 不合到 main（仅合到 test） |

### 2.2 矛盾解决

**矛盾 1**：实时推送 vs 服务端复杂度
- **决策**：P3-2 仅实现单进程内的 WebSocket 广播（用 `tokio::sync::broadcast`）
- **理由**：避免引入 Redis/NATS 依赖，简化 demo；多进程推送留 P4+

**矛盾 2**：JWT 鉴权 vs WebSocket 协议限制
- **决策**：通过 URL query 参数 `?token=xxx` 携带 JWT（与浏览器 WebSocket API 一致）
- **理由**：浏览器 WebSocket API 不支持自定义 header；query 参数是行业标准（如 Slack、Discord）

**矛盾 3**：主项目改造 vs 沙箱限制
- **决策**：主项目新增 `websocket/` 模块和路由，不修改现有通知 service
- **理由**：通过事件订阅机制，notification_service 发消息时**额外**触发 WebSocket 推送

#### 3.2.2 消息格式（JSON）

**服务端 → 客户端**：
```json
// 通知消息
{
  "type": "notification",
  "data": {
    "id": 42,
    "title": "订单已创建",

    "category": "order",
    "priority": 5,
    "created_at": "2026-06-17T10:30:00Z"
  }

// 心跳（pong）
{
  "type": "pong",
  "timestamp": 1718612400
}

// 错误
{
  "type": "error",
  "code": "INVALID_TOKEN",
  "message": "JWT 验证失败"
}
```

**客户端 → 服务端**：
```json

  "type": "ping",
  "timestamp": 1718612400
}

// 标记已读
{
  "type": "mark_as_read",
  "id": 42
}
```

### 3.3 后端实现要点

#### 3.3.1 WebSocket handler

```rust
// backend/src/websocket/notifications.rs

use axum::{
    extract::ws::{Message, WebSocket, WebSocketUpgrade},
    extract::Query,
    response::IntoResponse,
};
use tokio::sync::broadcast;

pub async fn ws_notifications_handler(
    ws: WebSocketUpgrade,
    Query(params): Query<HashMap<String, String>>,
) -> impl IntoResponse {
    // 1. 验证 JWT
    let token = params.get("token").ok_or(...)?;
    let auth = verify_jwt(token)?;
    let tenant_id = auth.tenant_id;
    let user_id = auth.user_id;

    // 2. 升级到 WebSocket
    ws.on_upgrade(move |socket| handle_socket(socket, tenant_id, user_id))
}
```

#### 3.3.2 连接管理

```rust
pub struct ConnectionManager {
    // 按 (tenant_id, user_id) 分组的发送者
```

## 六、风险与回滚

### 6.1 风险

| 风险 | 等级 | 缓解 |
|------|------|------|
| 沙箱 OOM 编译失败 | 高 | 仅 `cargo check --lib` |
| WebSocket 库不兼容 | 中 | 使用 `axum::extract::ws`（官方支持） |
| 前端 TypeScript 编译错误 | 中 | 严格类型 + 测试 |
| 主项目兼容性 | 低 | 仅新增模块，不修改现有代码 |

### 6.2 回滚

- 删除 `backend/src/websocket/` 模块
- 删除 `frontend/src/utils/websocket.ts`
- 移除路由配置
- 不影响 P0/P1/P2/P3-1 已合入代码
