## 五、推荐执行策略

### 5.1 P12 批 1 执行方案

1. **创建 `feature/p12-batch1` 分支**（不直接基于 main，先保持 main 干净）
2. **3 个子代理串行派发**（参照 P11 批 1 验证通过的模式）：
   - 子代理 1：P2-1 全部 5 PR（useTableApi 抽象 → 4 个页面迁移）
   - 子代理 2：B-type-check（CI 5 job + vue-tsc 集成）
   - 子代理 3：P2-2 性能优化（DB N+1 审计 + Redis 缓存）
3. **每个子代理输出**：1 个 feature 分支 + N 个 commit + N 个 PR
4. **主代理接手**：监控 CI、修复子代理遗漏（参照 P11 批 1 经验）
5. **每个 PR 工作流**：squash merge + delete_branch
6. **批 1 收尾**：CHANGELOG + MEMORY 同步

### 5.2 子代理工作约束

- ✅ 严格按本路线图任务定义执行
- ✅ 单个任务多文件时建议分多个 PR
- ✅ 命名 ≤ 9 英文字符
- ✅ 代码注释必须中文
- ❌ 禁止本地 `cargo build/clippy/test`（仅 CI 验证）
- ❌ 禁止删除其他任务的代码
- ⚠️ 本地 `cargo fmt` 仅在 fmt CI 失败时主代理临时使用

### 5.3 风险预警

| 风险 | 概率 | 缓解 |
|------|------|------|
| P2-1 5 PR 跨度大，子代理中途出错 | 中 | 拆为更小子代理（每个 PR 1 个子代理）|
| B-type-check 暴露大量 vue-tsc 错误 | 中 | 分批修复 + 短期豁免名单 |
| P2-2 性能优化改动 DB 层，影响广 | 中 | 先审计不改，写报告再优化 |
| 子代理 git workflow 遗漏 | 高 | 主代理监督每个 PR |
