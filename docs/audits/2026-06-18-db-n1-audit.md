## 三、修复优先级建议

| 优先级 | 位置数 | 建议策略 |
|--------|--------|----------|
| P0（高） | 4 | **预查询 + HashMap 缓存**（一次 `IN (...)` 查询，循环内 HashMap 查值）。本批缓存层（见第五节）可叠加做读路径降级 |
| P1（中） | 7 | 同上策略；当 N < 20 时可只加缓存层 |
| P2（低） | 5 | 暂不修复，列入下一批债务清理 |

> 完整 P0/P1 修复建议脚本：循环前置一次 `IN (?)` 查询 → 转 `HashMap<i32, Model>` → 循环内 `map.get(&id)` 替换 `find_by_id`。典型代码模板：

```rust
// 改造前
for item in &req.items {
    let obj = SomeEntity::find_by_id(item.obj_id).one(db).await?;
    process(obj);
}
// 改造后
let ids: Vec<i32> = req.items.iter().map(|i| i.obj_id).collect();
let objs = SomeEntity::find()
    .filter(SomeEntity::Column::Id.is_in(ids))
    .all(db).await?;
let map: HashMap<i32, SomeEntity::Model> =
    objs.into_iter().map(|o| (o.id, o)).collect();
for item in &req.items {
    process(map.get(&item.obj_id));
}
```

## 四、Redis 缓存层设计（P2-2 性能优化）

### 4.1 关键决策

| 决策 | 取值 | 理由 |
|------|------|------|
| 客户端 | `redis 0.27`（`tokio-comp` + `connection-manager`） | 异步 + 自动重连，符合 Tokio 运行时 |
| 键空间 | `tenant:{tenant_id}:{entity_type}:{entity_id}` | 强租户隔离，便于按业务前缀清理 |
| TTL 默认 | 300 秒（`CACHE_TTL_SECS` 可覆盖） | 兼顾新鲜度与命中率 |
| 配置 | 环境变量 `REDIS_URL` | 禁止硬编码；未设置时自动降级 |
| 失败策略 | **graceful degradation** | Redis 故障不影响业务路径 |
| 测试隔离 | `CacheBackend` trait + Mock | 单测不依赖真实 Redis |

### 4.2 模块结构

```
backend/src/cache/
├── mod.rs               # 暴露 CacheService / CacheBackend / CacheStats / RedisBackend
└── redis_client.rs      # 客户端封装 + 业务门面 + 单测
```

### 4.3 关键 API

```rust
// 业务门面
let cache = CacheService::from_env().await;          // 读 REDIS_URL / CACHE_TTL_SECS
let key = CacheService::build_key(tenant_id, "user", &id.to_string());
let user: Option<user::Model> = cache.get_json(&key).await;
cache.set_json(&key, &user, None).await;
cache.invalidate(&key).await;

// 统计
let (hits, misses, errors, fallbacks) = cache.stats().snapshot();
```

### 4.4 容错语义

| 异常场景 | 行为 | 统计 |
|----------|------|------|
| `REDIS_URL` 未设置 | `CacheService::disabled()`，所有操作走 `NullBackend` | `fallbacks +1` |
| Redis 连接失败 | 返回 `Self::disabled()`，业务继续 | 启动期一次性 warn |
| get 失败 | 返回 `None` 并记 `fallback`，业务回退到 DB | `errors +1`, `fallbacks +1` |
| set 失败 | warn 后吞掉，不影响主流程 | `errors +1` |
| 反序列化失败 | 视为 miss | `errors +1` |

## 五、示范 service 改造（user + product）

### 5.1 `user_service::find_by_id` 改造前后对比

**改造前**（直接查 DB）：

```rust
pub async fn find_by_id(&self, id: i32) -> Result<user::Model, AppError> {
    user::Entity::find_by_id(id)
        .one(self.db.as_ref())
        .await?
        .ok_or_else(|| AppError::not_found(format!("用户 ID {} 不存在", id)))
}
```

**改造后**（先缓存后 DB + 写时失效）：

```rust
pub fn with_cache(db: Arc<DatabaseConnection>, cache: Arc<CacheService>) -> Self {
    Self { db, cache: Some(cache) }
}

fn cache_key(user_id: i32) -> String {
    CacheService::build_key(0, "user", &user_id.to_string())  // 用户表为平台级
}

pub async fn find_by_id(&self, id: i32) -> Result<user::Model, AppError> {
    let key = Self::cache_key(id);
    if let Some(cache) = &self.cache {
        if let Some(cached) = cache.get_json::<user::Model>(&key).await {
            return Ok(cached);                              // 命中
        }
    }
    let user = user::Entity::find_by_id(id)                 // miss → 查 DB
        .one(self.db.as_ref()).await?
        .ok_or_else(|| AppError::not_found(format!("用户 ID {} 不存在", id)))?;
    if let Some(cache) = &self.cache {
        cache.set_json(&key, &user, None).await;            // 回写
    }
    Ok(user)
}
```

写路径同步接入 `invalidate`：
- `update_user` 末尾：`cache.invalidate(&Self::cache_key(user_id)).await`
- `update_last_login` 末尾同上
- `delete_user` 末尾同上

### 5.2 `product_service::get_product` 改造前后对比

**改造前**：

```rust
pub async fn get_product(&self, id: i32) -> Result<product::Model, AppError> {
    ProductEntity::find_by_id(id)
        .one(&*self.db)
        .await?
        .ok_or_else(|| AppError::not_found(format!("产品 ID {} 不存在", id)))
}
```

**改造后**：

```rust
fn cache_key(product_id: i32) -> String {
    CacheService::build_key(0, "product", &product_id.to_string())
}

pub async fn get_product(&self, id: i32) -> Result<product::Model, AppError> {
    let key = Self::cache_key(id);
    if let Some(cache) = &self.cache {
        if let Some(cached) = cache.get_json::<product::Model>(&key).await {
            return Ok(cached);
        }
    }
    let product = ProductEntity::find_by_id(id)
        .one(&*self.db).await?
        .ok_or_else(|| AppError::not_found(format!("产品 ID {} 不存在", id)))?;
    if let Some(cache) = &self.cache {
        cache.set_json(&key, &product, None).await;
    }
    Ok(product)
}
```

写路径同步接入 `invalidate`：
- `update_product` 末尾
- `delete_product` 末尾

### 5.3 改造收益（定性）

| 场景 | 改造前 | 改造后（命中率 80%） |
|------|--------|---------------------|
| `find_by_id` 用户详情 | 每次 1 次 DB | 命中时 0 次 DB；miss 仍 1 次 |
| `get_product` 产品详情 | 每次 1 次 DB | 同上 |
| `user_handler` 详情页 | 5–10 次 DB | 命中后 0–2 次 DB |

> 关键：写路径已显式 `invalidate`，不存在"读陈旧"风险；TTL 300 秒仅在写漏时兜底。
