# 秉羲面料管理优化与功能扩展 - 产品需求文档

## 概述
- **摘要**：对秉羲ERP系统进行全面优化，包括代码质量提升、安全强化、访问控制完善，以及新增供应商产品管理和销售单自动转采购单功能。
- **目的**：提升系统安全性、代码质量和用户体验，实现供应商产品与内部产品的映射管理，以及销售订单到采购订单的自动化转换流程。
- **目标用户**：面料行业企业的采购人员、销售人员、仓库管理人员、财务人员及系统管理员。

## 目标
- 强化系统安全性，确保未登录用户只能访问登录页面
- 实现完善的前后端访问控制，防止后端被直接访问
- 建立基于角色的功能权限和数据权限体系
- 实现供应商产品色号与内部产品色号的映射管理
- 实现销售订单审批后自动生成采购订单的功能
- 优化代码性能、可读性、简洁性
- 确保功能完整性和唯一性

## 非目标（范围外）
- 移动端应用开发
- 第三方系统集成（如财务软件接口）
- 多语言国际化支持
- 数据迁移工具开发

## 背景与上下文
- **现有系统**：秉羲面料管理是一个基于Rust语言开发的面料行业ERP系统，前端使用Yew框架，后端使用Axum框架
- **技术栈**：Rust 2021、PostgreSQL、SeaORM、JWT认证
- **当前问题**：
  - 前端缺少路由守卫，未登录可访问其他页面
  - 后端可被直接访问，缺少访问控制
  - 权限控制不完善，缺少数据权限
  - 供应商产品管理功能缺失
  - 销售转采购流程需要手动操作
  - 代码存在重复、性能问题

## 功能需求

### FR-1：前端路由保护
- 未登录用户只能访问登录页面
- 登录后才能访问其他功能页面
- Token过期自动跳转登录页

### FR-2：后端访问控制
- CORS配置限制只允许前端域名访问
- Nginx配置阻止后端直接访问
- 请求头验证确保请求来自前端

### FR-3：认证中间件增强
- 公开路径配置化管理
- 请求来源验证
- 认证失败日志记录

### FR-4：权限控制体系
- 基于角色的功能权限控制
- 基于角色的数据权限控制
- 权限验证中间件

### FR-5：供应商产品管理
- 供应商可提供多个产品
- 每个产品可有多个色号
- 供应商产品色号与内部产品色号映射

### FR-6：销售转采购自动转换
- 销售订单审批后自动触发
- 按供应商拆分采购订单
- 自动选择默认供应商
- 产品色号自动转换

### FR-7：代码性能优化
- 数据库查询优化（索引、N+1问题）
- 缓存策略实现
- 前端性能优化

### FR-8：代码简洁性优化
- 消除重复代码
- 简化复杂函数
- 移除未使用代码

### FR-9：功能完整性和唯一性
- 补全缺失的CRUD操作
- 统一API响应格式
- 消除重复功能接口

## 非功能需求

### NFR-1：安全性
- 所有API接口必须经过认证
- 敏感数据加密存储
- 防止SQL注入、XSS攻击
- 操作日志记录

### NFR-2：性能
- API响应时间 < 200ms（95%请求）
- 数据库查询优化，避免N+1问题
- 前端首屏加载时间 < 3s

### NFR-3：可维护性
- 代码注释覆盖率 > 30%
- 遵循Rust代码规范
- 模块化设计，单一职责

### NFR-4：可扩展性
- 权限系统支持自定义角色
- 供应商产品映射支持扩展属性
- 采购转换规则可配置

## 约束条件
- **技术约束**：使用Rust语言、Yew框架、Axum框架、PostgreSQL数据库
- **业务约束**：面料行业特定需求（色号、批次、双计量单位）
- **依赖约束**：现有数据库结构、现有API接口

## 假设
- 用户已熟悉系统基本操作
- 网络环境稳定
- 服务器资源充足（至少2核CPU、2GB内存）

## 验收标准

### AC-1：前端路由保护
- **Given**：用户未登录
- **When**：用户尝试访问除登录页以外的任何页面
- **Then**：系统自动重定向到登录页面
- **验证方式**：`programmatic`

### AC-2：后端访问控制 - CORS
- **Given**：后端服务已启动
- **When**：来自非允许域名的跨域请求
- **Then**：请求被拒绝，返回CORS错误
- **验证方式**：`programmatic`

### AC-3：后端访问控制 - 请求头验证
- **Given**：后端服务已启动
- **When**：请求缺少X-Requested-With请求头
- **Then**：请求被拒绝，返回403错误
- **验证方式**：`programmatic`

### AC-4：认证中间件增强
- **Given**：用户发送API请求
- **When**：Token无效或过期
- **Then**：返回401错误并记录日志
- **验证方式**：`programmatic`

### AC-5：功能权限控制
- **Given**：用户已登录但无某功能权限
- **When**：用户尝试访问该功能
- **Then**：系统拒绝访问并提示权限不足
- **验证方式**：`programmatic`

### AC-6：数据权限控制
- **Given**：用户只有本人数据权限
- **When**：用户尝试访问其他用户的数据
- **Then**：系统拒绝访问
- **验证方式**：`programmatic`

### AC-7：供应商产品管理 - 创建映射
- **Given**：供应商和内部产品已存在
- **When**：管理员创建供应商产品色号映射
- **Then**：映射关系成功保存，可查询和修改
- **验证方式**：`programmatic`

### AC-8：供应商产品管理 - 查询映射
- **Given**：供应商产品映射已存在
- **When**：用户查询供应商提供的产品列表
- **Then**：系统返回该供应商的所有产品及色号映射
- **验证方式**：`programmatic`

### AC-9：销售转采购 - 自动触发
- **Given**：销售订单已创建且有产品色号映射
- **When**：销售订单审批通过
- **Then**：系统自动生成采购订单（按供应商拆分）
- **验证方式**：`programmatic`

### AC-10：销售转采购 - 产品转换
- **Given**：销售订单包含内部产品色号
- **When**：生成采购订单
- **Then**：采购订单中的产品色号自动转换为供应商产品色号
- **验证方式**：`programmatic`

### AC-11：销售转采购 - 供应商选择
- **Given**：产品有多个供应商映射
- **When**：生成采购订单
- **Then**：系统自动选择默认供应商（优先级最高或最近使用）
- **验证方式**：`programmatic`

### AC-12：代码性能优化
- **Given**：系统正常运行
- **When**：执行数据库查询
- **Then**：无N+1查询问题，查询使用索引
- **验证方式**：`programmatic`

### AC-13：代码简洁性
- **Given**：代码库
- **When**：执行代码审查
- **Then**：无重复代码块，无未使用的代码，函数复杂度合理
- **验证方式**：`human-judgment`

### AC-14：功能完整性
- **Given**：系统所有模块
- **When**：检查每个模块的CRUD操作
- **Then**：所有模块都有完整的CRUD操作
- **验证方式**：`programmatic`

### AC-15：API响应格式统一
- **Given**：所有API接口
- **When**：调用任意API
- **Then**：响应格式统一为 { code, message, data }
- **验证方式**：`programmatic`
