### 1.2 复审通用流程

```
复审阶段（自动执行）
  ├── 1. 按维度逐项扫描代码库
  ├── 2. 每个维度生成独立报告（保存到 docs/audits/）
  ├── 3. 按优先级排序修复队列（P0→P1→P2→P3）
  └── 4. 生成总复审报告
       │
       ▼
修复阶段（复审完成后自动开始）
  ├── 5. 按规则 13 流程执行：建分支→修改→commit→push→PR→CI→merge→下一批
  ├── 6. 每批 5-8 文件，CI 全绿后自动进入下一批
  ├── 7. 每批更新 doto.md 进度 + CHANGELOG.md 一句话总结
  └── 8. 所有批次完成后进行回归验证
       │
       ▼
闭环验证
  ├── 9. 确认无新增问题
  ├── 10. baseline 警告数同步减少
  └── 11. 触发下一轮复审（如 v13→v14）
```

### 1.3 优先级分类标准

| 优先级 | 定义 | 处理时机 | 示例 |
|--------|------|----------|------|
| **P0** | 阻塞级：核心业务链路断裂、数据丢失风险、安全漏洞 | 立即修复，最高优先 | 销售订单审批后未触发库存预留 |
| **P1** | 高优先级：跨模块一致性缺失、事件闭环断裂、配置陷阱 | 当前阶段内修复 | 客户主数据变更未同步关联单据 |
| **P2** | 中优先级：代码质量、类型安全、日志规范 | 当前阶段内修复 | 前端 unknown 类型细化 |
| **P3** | 低优先级：命名规范、注释完善、代码风格 | 后续阶段批量修复 | 变量命名不统一 |

#### 2.2.1 四层级联关系（所有维度的基础）

```
面料（Fabric/Product）
  └── 颜色（Color）              ← 一个面料有多个颜色
       ├── 色号（ColorNo）        ← 颜色唯一编码
       ├── 色卡（ColorCard）      ← 颜色实物样卡
       └── 缸号（DyeLotNo）       ← 每个颜色每批次染色有独立缸号
            ├── 批号（BatchNo）    ← 同一缸号下可分多个批次（匹号）
            ├── 库存数量          ← 按缸号/批号独立核算
            ├── 质检结果          ← 按缸号/批号独立检验
            ├── 成本单价          ← 按缸号/批号独立核算（实际成本）
            └── 库位              ← 按缸号分区存放
```

#### 2.2.2 关键业务约束（必须强制执行）

- **匹号唯一约束**：**同一缸号（dye_lot_no）下不能有相同的匹号（batch_no）**
  - DB 层：`UNIQUE(dye_lot_no, batch_no)` 唯一索引
  - Service 层：创建/更新时校验 (dye_lot_no, batch_no) 组合唯一
  - 全局适用：库存表、入库表、发货表、退货表、盘点表等所有含 batch_no 的表均需校验
  - 业务语义：一个缸号代表一次染色，同一缸内分多匹出布，每匹有独立匹号；匹号重复会导致库存和成本核算混乱
- **面料-颜色关联约束**：`UNIQUE(product_id, color_id)`
- **缸号-颜色关联约束**：`UNIQUE(color_id, dye_lot_no)`
- **库存四维标识**：库存唯一标识 = product_id + color_id + dye_lot_no + batch_no（联合唯一索引）

#### 2.2.3 复用现有功能原则（v14 复审最高准则）

- 修复前必须调研现有实现，禁止重复造轮子，优化而非新建
- 复用评估清单：现有 service/composable/component/utils/DB 表/API/事件是否已实现类似功能
- 共享工具优先：`utils/dual_unit_converter`、`utils/number_generator`、`utils/pagination` 等
- 前端 composable 优先：`useTableApi`、`useFormApi` 等已封装的 composable
- 后端 service 优先：`voucher_service`、`audit_log_service`、`event_bus` 等已封装的 service
- DB 模型扩展优先：新增字段优先扩展已有表，而非新建表
- API 契约兼容：修复时优先保持 API 返回格式兼容，避免破坏前端已有调用

## 四、clippy baseline 管理规则

### 4.1 baseline 机制

- **文件位置**：[backend/.clippy-baseline.txt](file:///workspace/backend/.clippy-baseline.txt)
- **比较逻辑**：CI 提取 `warning:`/`error:` 开头的摘要行，`comm -23` 比较当前与基线，新增警告 = CI 失败
- **bootstrap 模式**：基线文件不存在时，CI 自动用当前警告建立基线，不阻塞
- **strict 模式**：基线文件存在时，新增警告阻塞 CI

### 4.2 baseline 更新规则

| 场景 | 操作 | 执行者 |
|------|------|--------|
| 基线文件不存在 | CI 自动创建并提交（bootstrap 模式） | CI 自动 |
| main 分支修复了历史警告 | CI 自动刷新基线（移除已修复条目） | CI 自动 |
| PR 分支修复了历史警告 | 合并到 main 后 CI 自动刷新 | CI 自动 |
| 基线文件被意外破坏 | 删除文件，让 CI bootstrap 重建 | 手动 |

### 4.3 历史教训

- **批次 402 错误**：手动创建基线文件时只写了 1 行，导致 117 个历史警告被误判为新警告，CI clippy 失败
- **批次 406 修复**：删除错误基线文件，CI bootstrap 模式自动重建 180 行完整基线
- **教训**：基线文件应由 CI 自动生成，不要手动创建
