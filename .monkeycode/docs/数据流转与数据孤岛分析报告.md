## 二、已打通的数据流转链路

### 2.1 核心业务流程

#### 采购到付款流程 (Procure-to-Pay)
```
供应商 → 采购订单 → BPM审批 → 入库确认 → 库存更新
                                    ↓
                              自动生成应付账单 → 付款申请 → 付款 → 核销
```

**数据流转路径**:
- `suppliers` → `purchase_orders` → `purchase_order_items`
- `purchase_orders` → `bpm_process_instance` (审批)
- `purchase_receipt` → `purchase_receipt_item` → `inventory_stocks` (入库)
- `purchase_receipt` → `ap_invoice` (自动)
- `ap_invoice` → `ap_payment_request` → `ap_payment` → `ap_verification`

#### 订单到收款流程 (Order-to-Cash)
```
客户 → 销售订单 → BPM审批 → 订单完成 → 自动生成应收账款
                                    ↓
                              发货 → 库存扣减 → 收款 → 对账
```

**数据流转路径**:
- `customers` → `sales_orders` → `sales_order_items`
- `sales_orders` → `bpm_process_instance` (审批)
- `sales_delivery` → `sales_delivery_item` → `inventory_stocks` (出库)
- `sales_orders` → `ar_invoices` (自动)
- `ar_invoices` → `ar_collections` → `ar_reconciliations`

#### 退货流程
```
采购退货 → 库存扣减 + 应付红字冲销
销售退货 → 库存更新 + 应收调整
```

### 2.2 事件驱动的异步数据流

| 事件 | 触发条件 | 处理逻辑 |
|------|----------|----------|
| `BpmProcessFinished` | 采购订单审批完成 | 更新订单状态为"已审批" |
| `BpmProcessFinished` | 销售订单审批完成 | 更新订单状态为"已审批" |
| `PurchaseReceiptCompleted` | 入库确认完成 | 更新采购订单收货状态 |
| `PaymentCompleted` | 付款完成 | 更新应付发票已付金额 |
| `CollectionCompleted` | 收款完成 | 更新应收发票已收金额 |

## 三、数据孤岛问题

### 3.1 孤立模块列表

以下模块与其他业务模块缺乏数据联动：

| 模块 | 服务文件 | 孤立表现 | 影响 |
|------|----------|----------|------|
| **CRM模块** | `crm_service` | 不调用任何其他服务 | 客户数据无法自动流向销售模块 |
| **产能分析** | `capacity_service` | 独立运行 | 生产排程无法获取产能数据 |
| **缺料预警** | `material_shortage_service` | 独立运行 | 无法自动触发采购申请 |
| **财务分析** | `financial_analysis_service` | 独立运行 | 需要手动获取业务数据 |
| **库存预警** | `inventory_stock_service` | 低库存预警不联动 | 无法自动触发补货建议 |
| **报表引擎** | `report_engine_service` | 独立运行 | 缺乏标准化数据聚合接口 |
| **客户信用** | `customer_credit_service` | 独立运行 | 信用评估不与销售订单联动 |
| **预算管理** | `budget_management_service` | 独立运行 | 预算控制不与采购/付款联动 |

### 3.3 关键数据孤岛风险

#### 风险1: CRM与销售模块脱节
**问题**: CRM的客户数据无法自动流向销售模块
**影响**: 销售人员需要手动关联客户信息
**建议**: 
- 在创建销售订单时自动关联CRM客户
- 销售订单完成后自动更新CRM客户状态

#### 风险2: 库存预警与采购模块脱节
**问题**: 低库存预警无法自动触发采购申请
**影响**: 需要人工监控库存并手动创建采购申请
**建议**:
- 实现库存预警事件
- 预警触发时自动生成采购建议

#### 风险3: 产能分析与生产排程脱节
**问题**: 生产排程无法获取产能数据
**影响**: 排程结果可能超出实际产能
**建议**:
- 生产排程时自动查询产能数据
- 实现产能负荷预警

#### 风险4: 财务分析与业务数据脱节
**问题**: 财务分析需要手动获取业务数据
**影响**: 分析结果可能不准确或滞后
**建议**:
- 实现财务指标自动计算
- 建立业务数据到财务数据的自动聚合

## 四、数据流转问题

### 4.1 事件总线绕过Service层

`event_bus.rs`中有3处直接操作数据库模型，绕过了Service层的业务逻辑：

```rust
// 第93行：直接更新 purchase_order 状态
// 第108行：直接更新 ap_invoice 状态
// 第127行：直接更新 ar_invoice 状态
```

**风险**:
- 绕过业务逻辑校验
- 绕过审计日志记录
- 可能导致数据一致性问题

**建议**:
- 改为调用对应的Service方法
- 或在Service层提供专门的状态更新方法

### 4.2 缺少统一的事件机制

当前事件驱动主要通过`EventBus`实现，但存在以下问题：
- 事件定义不统一
- 事件处理器分散
- 缺少事件溯源机制

**建议**:
- 统一事件定义和命名规范
- 建立事件注册和发现机制
- 考虑引入事件溯源

## 五、改进建议

### 5.1 短期改进（1-2周）

1. **修复事件总线问题**
   - 将直接模型操作改为调用Service方法
   - 添加审计日志记录

2. **建立库存预警联动**
   - 实现低库存事件
   - 事件触发时生成采购建议

3. **完善CRM与销售联动**
   - 销售订单自动关联CRM客户
   - 订单完成更新CRM状态

### 5.2 中期改进（1-2月）

1. **建立统一事件机制**
   - 定义标准事件接口
   - 实现事件注册和发现
   - 添加事件溯源支持

2. **打通产能与排程**
   - 生产排程查询产能数据
   - 实现产能负荷预警

3. **实现财务指标自动计算**
   - 建立业务数据聚合接口
   - 实现财务指标自动更新

### 5.3 长期改进（3-6月）

1. **建立数据中台**
   - 统一数据访问层
   - 实现数据标准化
   - 提供数据聚合服务

2. **引入领域事件**
   - 按业务领域定义事件
   - 实现事件驱动架构
   - 支持事件回放和重试

3. **建立数据质量监控**
   - 实现数据一致性检查
   - 建立数据质量指标
   - 自动告警异常数据

## 六、总结

### 6.1 整体评估

- **核心业务流程**: 数据流转完整，外键约束覆盖良好
- **辅助功能模块**: 存在数据孤岛，需要应用层维护一致性
- **事件驱动机制**: 基本可用，但存在绕过Service层的问题

| 指标 | 当前值 | 目标值 |
|------|--------|--------|
| 外键覆盖率 | 66% | 80% |

| 数据孤岛模块数 | 8个 | 3个 |
| 事件驱动调用数 | 6处 | 15处 |
