# 批量修复计划

## 一、后端修复

### 1.1 打印功能实现（高优先级）

创建通用打印服务，支持所有单据类型：
- 销售订单、销售合同、销售退货
- 采购订单、采购合同、采购入库
- 库存单据（调拨、盘点、调整）
- 财务凭证

**实现文件**:
- `backend/src/services/print_service.rs` - 通用打印服务
- `backend/src/handlers/print_handler.rs` - 打印接口
- 所有相关 routes 添加 `/:id/print` 路由

### 1.2 响应格式统一

确保所有 list 接口返回：
```json
{
  "code": 200,
  "data": {
    "items": [...],
    "total": 100
  }
}
```

还是返回：
```json
{
  "code": 200,
  "data": [...]
}
```

需要统一，前端已经适配 `items || data || []` 的模式。

### 1.3 批量操作接口

添加批量删除、批量审核等接口：
- `POST /bulk-delete`
- `POST /bulk-approve`
- `POST /bulk-reject`

## 二、前端修复

### 2.1 打印按钮实现

所有页面的打印按钮统一调用 `/xxx/:id/print` 接口，返回 PDF 或新窗口打开打印页面。

### 2.2 保存/编辑错误修复

检查所有表单：
- 必填项验证
- 字段格式验证
- API 调用参数正确性

### 2.3 404 错误修复

检查所有 API 路径是否匹配后端 route。

## 三、测试验证

1. 所有新增功能测试
2. 所有保存/编辑功能测试
3. 所有打印功能测试
4. 回归测试（确保不影响已有功能）

## 四、实施步骤

1. 创建通用打印服务和 handler
2. 为所有需要的模块添加打印路由
3. 前端统一添加打印按钮处理逻辑
4. 测试所有保存/编辑功能，修复报错
5. 提交代码 -> CICD 构建 -> 部署 -> 验证

## 五、预计时间

- 后端打印功能：2 小时
- 前端打印适配：1 小时
- 保存/编辑修复：2 小时
- 测试验证：1 小时

**总计：约 6 小时**
