## 一、P0 严重问题（2 项）

### P0-1：backup.rs cmd_backup 中 pg_dump 失败不返回 false

**文件**：[backend/src/cli/util/backup.rs](file:///workspace/backend/src/cli/util/backup.rs#L87-L96)

**问题**：
```rust
match run_cmd("pg_dump", &[...]) {
    Ok(_) => println!("[OK] 数据库备份完成"),
    Err(e) => println!("[ERROR] 数据库备份失败: {}", e),  // 仅打印，未 return false
}
```

**影响**：
- L4 修复（v8 复审）将 `cmd_backup` 返回类型改为 `bool`，意图让 `upgrade` 根据备份结果中止升级
- 但 pg_dump 失败时函数仍返回 `true`，导致 upgrade 拿到"备份成功"假象继续部署
- 数据库备份失败 + 部署失败 → 无法回滚到正确备份，数据丢失风险

**修复方案**：pg_dump 失败时 `return false`（数据库是核心数据，备份失败必须中止）

### P0-2：system_update_service.rs L7 目录权限掩码未应用

**文件**：[backend/src/services/system_update_service.rs](file:///workspace/backend/src/services/system_update_service.rs#L424-L457)

**问题**：
```rust
if file.name().ends_with('/') {
    fs::create_dir_all(&outpath)?;  // 目录分支：未设置权限
} else {
    // ... fs::File::create(&outpath) ...
    let safe_mode = if outpath.is_dir() {  // 这里永远为 false（文件刚创建）
        mode & 0o755
    } else {
        mode & 0o600  // 文件分支：生效
    };
    // 设置权限
}
```

**影响**：
- `outpath.is_dir()` 在文件分支内永远为 false（文件刚通过 `fs::File::create` 创建）
- 目录分支（`file.name().ends_with('/')`）未设置任何权限，目录权限由 umask 决定
- 恶意 zip 可设置目录的 SUID/SGID/sticky bit，导致权限提升
- L7 修复（v8 复审）的"目录使用 0o755"目标完全未达成

**修复方案**：将权限设置分别移到两个分支内
- 目录分支：`fs::create_dir_all` 后设置 `mode & 0o755`
- 文件分支：`fs::File::create` 后设置 `mode & 0o600`

## 二、高危问题（2 项）

### H-1：upgrade.rs Tar Slip 路径穿越（固定路径 + 校验范围不足 + 先解压后校验）

**文件**：[backend/src/cli/util/upgrade.rs](file:///workspace/backend/src/cli/util/upgrade.rs#L216-L227)

**问题**：
```rust
// 1. 直接解压到 /tmp（固定路径，存在符号链接竞争）
if let Err(e) = run_cmd("tar", &["-xzf", package, "-C", "/tmp"]) { ... }

// 2. 仅校验 /tmp/bingxi-erp 子目录（攻击者可写入 /tmp 其他位置）
let extract_dir = "/tmp/bingxi-erp";
if let Err(e) = validate_extracted_path(extract_dir) { ... }
```

**影响**：
- 固定路径 `/tmp` 存在符号链接竞争（TOCTOU）：攻击者预先在 `/tmp/bingxi-erp` 创建符号链接指向 `/etc`
- 校验仅覆盖 `/tmp/bingxi-erp`，但 tar 可能解压出 `/tmp/其他文件` 直接绕过校验
- 先解压后校验：恶意文件在校验前已写入磁盘，可能触发漏洞
- 对比 backup.rs 的 M4 修复（先 `tar -tf` 校验再解压 + UUID 随机目录），upgrade.rs 修复不完整

**修复方案**（对齐 backup.rs 方案）：
1. 使用 UUID 随机目录：`/tmp/bingxi_upgrade_{uuid}`
2. 先 `tar -tf` 列出内容校验路径（含 `..`/绝对路径检测）
3. 再 `tar -xzf` 解压到随机目录
4. 解压后 `validate_extracted_path` 二次校验

### H-2：admin.rs bingxi 自身命令行仍暴露密码

**文件**：[backend/src/cli/util/admin.rs](file:///workspace/backend/src/cli/util/admin.rs)

**问题**：v8 H4 修复仅解决 python 子进程层面的密码泄露（`-p MySecret` 通过 `process::Command::arg` 传递），但 bingxi 自身命令行（如 `bingxi admin reset-password -p MySecret`）仍通过 `clap` 参数接收密码，会出现在 `ps`/`procfs` 中

**影响**：
- 同机其他用户可通过 `ps aux | grep bingxi` 看到密码
- 容器内 `/proc/<pid>/cmdline` 可读
- v8 H4 修复不彻底

**修复方案**：密码参数改为 `--password-stdin`（从 stdin 读取），或 `BINGXI_ADMIN_PASSWORD` 环境变量

## 三、中危问题（5 项）

### M-1：system_update_service.rs fetch_latest_release 未用 resolve_to_addrs

**文件**：[backend/src/services/system_update_service.rs](file:///workspace/backend/src/services/system_update_service.rs#L634-L663)

**问题**：`fetch_latest_release` 仅添加 `redirect(Policy::limited(3))`，未使用 `validate_url_and_resolve` + `resolve_to_addrs` 防 DNS Rebinding

**对比**：`download_update`（L741-755）已使用 `validate_url_and_resolve` + `resolve_to_addrs`，但 `fetch_latest_release` 未对齐

**修复方案**：对齐 `download_update` 的 SSRF 防护

### M-2：system_update_service.rs download_path 未校验 asset.name

**文件**：[backend/src/services/system_update_service.rs](file:///workspace/backend/src/services/system_update_service.rs#L738)

**问题**：
```rust
let download_path = download_dir.join(&asset.name);  // asset.name 来自 GitHub API，但未校验
```

**影响**：恶意 GitHub Release（若账号被入侵）可设置 `asset.name = "../../../etc/cron.d/evil"`，导致路径穿越

**修复方案**：校验 `asset.name` 只包含安全字符（字母数字.-_），拒绝 `/`、`..`、绝对路径

### M-3：webhook_handler.rs retry_webhook 缺失限流

**文件**：[backend/src/handlers/webhook_handler.rs](file:///workspace/backend/src/handlers/webhook_handler.rs)

**问题**：`test_webhook` 已在批次 306 接入 `check_rate_limit`（M6 修复），但 `retry_webhook` 端点未限流

**影响**：攻击者可高频调用 `retry_webhook` 触发大量出站 HTTP 请求，导致 SSRF 放大攻击

**修复方案**：`retry_webhook` 接入 `check_rate_limit`（与 `test_webhook` 共用 `WEBHOOK_TEST_LIMITER` 或独立限流器）

### M-4：webhook_handler.rs 所有 webhook 端点 IDOR，无所有权校验

**文件**：[backend/src/handlers/webhook_handler.rs](file:///workspace/backend/src/handlers/webhook_handler.rs)

**问题**：webhook 端点（如 `delete_webhook`、`update_webhook`、`retry_webhook`）仅校验用户登录，未校验 webhook 是否属于该用户

**影响**：IDOR（Insecure Direct Object Reference）— 用户 A 可删除/修改/重试用户 B 的 webhook

**修复方案**：所有 webhook 操作前校验 `webhook.user_id == auth.user_id`

### M-5：elastic.rs ES base_url 未做 SSRF 校验

**文件**：[backend/src/services/elastic.rs](file:///workspace/backend/src/services/elastic.rs)

**问题**：Elasticsearch 客户端构建时直接使用配置的 `base_url`，未做 SSRF 校验

**影响**：若管理员误配置或被攻击者篡改 `base_url` 为内网地址，可探测内网

**修复方案**：对齐 `validate_url_and_resolve` 校验 `base_url`
