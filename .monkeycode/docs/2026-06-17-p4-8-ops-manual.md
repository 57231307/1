```bash
  name: letsencrypt-prod
```

## 4. 备份与恢复

### 4.1 备份策略

| 类型 | 频率 | 保留 | 工具 | 大小预估 |
|------|------|------|------|---------|
| 数据库全量 | 每日 02:00 | 30 天 | `pg_dump` | 1-5 GB |
| 数据库增量 | 每小时 | 7 天 | WAL 流复制 | 50-200 MB/h |
| 配置文件 | 每次变更 | 永久 | Git | < 10 MB |
| 镜像备份 | 每次发布 | 90 天 | 阿里云 ACR | 500 MB/版本 |
| Redis 快照 | 每日 03:00 | 7 天 | `BGSAVE` | 200 MB |

### 4.2 备份脚本

`/opt/bingxi/backup/db-backup.sh`：

```bash
#!/bin/bash
set -euo pipefail

BACKUP_DIR=/data/backup/db/$(date +%Y%m%d_%H%M%S)
OSS_BUCKET=oss://bingxi-backup/db
RETENTION_DAYS=30
LOG_FILE=/var/log/bingxi/backup.log

mkdir -p "$BACKUP_DIR"
echo "[$(date)] Start backup" >> "$LOG_FILE"

# 1. 全量备份
pg_dump -Fc \
  -h "${DB_HOST:-postgres-primary}" \
  -U "${DB_USER:-erp}" \
  "${DB_NAME:-bingxi_erp}" \
  > "$BACKUP_DIR/db.dump"

# 2. 压缩
gzip "$BACKUP_DIR/db.dump"

# 3. 上传到 OSS
ossutil cp "$BACKUP_DIR/db.dump.gz" \
  "${OSS_BUCKET}/$(date +%Y%m%d)/" \
  >> "$LOG_FILE" 2>&1

# 4. 清理本地过期备份
find /data/backup/db -mtime +${RETENTION_DAYS} -delete

echo "[$(date)] Backup completed: $BACKUP_DIR/db.dump.gz" >> "$LOG_FILE"
```

### 4.3 定时任务

```bash
# 添加到 crontab
cat <<EOF | sudo crontab -
# 每日 02:00 全量备份
0 2 * * * /opt/bingxi/backup/db-backup.sh

# 每 6 小时健康检查
0 */6 * * * /opt/bingxi/backup/health-check.sh
EOF
```

### 4.4 恢复流程

#### 4.4.1 全量恢复

```bash
# 1. 停止应用（避免脏写）
kubectl scale deployment erp --replicas=0 -n erp

# 2. 下载备份
ossutil get oss://bingxi-backup/db/20260617/db.dump.gz

# 3. 解压
gunzip db.dump.gz

# 4. 恢复
pg_restore -h postgres-primary -U erp -d bingxi_erp --clean --if-exists db.dump

# 5. 启动应用
kubectl scale deployment erp --replicas=2 -n erp

# 6. 验证
curl http://erp.bingxi.example.com/api/health
```

#### 4.4.2 PITR（时间点恢复）

```bash
# 1. 找到目标时间
# 假设要恢复到 2026-06-17 14:30:00

# 2. 恢复基础备份
pg_restore -h postgres-standby -U erp -d bingxi_erp \
  --clean --if-exists base.dump

# 3. 应用 WAL 到目标时间
pg_waldump --start=.../0000000100000000000000A0 \
  | psql -h postgres-standby -U erp -d bingxi_erp

# 4. 验证数据
psql -c "SELECT count(*) FROM orders WHERE created_at > '2026-06-17 14:00:00';"
```

## 5. 故障排查

### 5.1 故障排查总览

| 症状 | 可能原因 | 排查命令 |
|------|---------|---------|
| Pod 一直 Pending | 节点资源不足 / PVC 等待 | `kubectl describe pod` |
| Pod CrashLoopBackOff | 镜像拉取失败 / 配置错误 | `kubectl logs` |
| 5xx 错误率高 | DB 故障 / 慢查询 / 资源耗尽 | 见 5.2 |
| 响应慢 | 慢查询 / 网络 / 缓存失效 | 见 5.3 |
| 登录失败 | DB 不可用 / JWT 密钥错误 | 见 5.4 |

### 5.2 5xx 错误率高

#### 5.2.1 排查步骤

```bash
# 1. 确认故障范围
kubectl get pods -n erp

# 2. 查看错误日志
kubectl logs -n erp -l app=bingxi-erp --tail=200 | grep ERROR

# 3. 查看慢查询
# Loki: {app="bingxi-erp"} |= "slow_query"

# 4. 查看 DB 状态
psql -h postgres-primary -U erp -c "SELECT count(*) FROM pg_stat_activity;"

# 5. 查看连接池
curl http://localhost:8080/api/metrics | grep erp_db_pool
```

#### 5.2.2 常见根因

| 根因 | 表现 | 解决方案 |
|------|------|---------|
| DB 连接池耗尽 | `erp_db_pool_overflow_total` 增长 | 增加 `DATABASE_POOL_SIZE` |
| 慢查询拖垮 | `erp_slow_queries_total` 增长 | 添加索引 / 优化 SQL |
| 内存溢出 | OOMKilled | 增加内存限制 / 排查泄漏 |
| 磁盘满 | `DiskSpaceLow` 告警 | 清理日志 / 扩容 |

### 5.3 响应慢

#### 5.3.1 排查步骤

```bash
# 1. 查看 P99 延迟
curl http://localhost:8080/api/metrics | grep http_request_duration

# 2. 慢查询分析
# Loki: {app="bingxi-erp"} |= "slow_query" | json | elapsed_ms > 1000

# 3. DB 慢查询
psql -c "SELECT pid, query, state, wait_event FROM pg_stat_activity WHERE state != 'idle';"

# 4. EXPLAIN 分析
psql -c "EXPLAIN ANALYZE SELECT ...;"
```

#### 5.3.2 优化手段

- 添加缺失索引
- 重写 N+1 查询
- 启用查询缓存
- 增加只读副本

### 5.4 登录失败

```bash
# 1. 查看登录失败次数
curl http://localhost:8080/api/metrics | grep erp_login_attempts

# 2. 检查 JWT 密钥

# 3. 验证密码哈希
```


## 7. 安全加固

### 7.1 认证与授权

- **JWT Token**：HS256，2 小时过期，refresh token 7 天
- **多因子认证**：TOTP 可选（管理后台强制）
- **密码策略**：8+ 位 + 大小写 + 数字 + 特殊字符
- **密码历史**：5 次内不可复用
- **账户锁定**：5 次失败锁 30 分钟

### 7.2 API 安全

| 措施 | 实现 | 位置 |
|------|------|------|
| 速率限制 | 令牌桶 60 req/min | `middleware/rate_limit.rs` |
| SQL 注入审计 | URL 模式匹配 | `middleware/sql_injection_audit.rs` |
| CSP | 响应头注入 | `middleware/csp.rs` |
| HSTS | 1 年 + 子域 | `security_headers.rs` |
| X-Frame-Options | DENY | `security_headers.rs` |
| CORS | 限定白名单 | `main.rs` |

### 7.3 数据安全

- **传输加密**：TLS 1.3（强制）
- **存储加密**：敏感字段 AES-256-GCM
- **密码哈希**：Argon2id
- **备份加密**：OSS 服务端加密
- **密钥管理**：K8s Secret + 定期轮换

### 7.4 网络安全

- **VPC 隔离**：应用 / DB / Redis 独立子网
- **安全组**：仅开放 443 / 80
- **WAF**：阿里云 WAF 防护 SQL 注入 / XSS
- **DDoS 防护**：高防 IP

### 7.5 审计

- **操作日志**：所有写操作记录到 `operation_logs` 表

## 9. 容量规划

### 9.1 当前规模（2026-06）

| 资源 | 当前 | 6 个月 | 12 个月 |
|------|------|--------|---------|
| 租户数 | 50 | 200 | 500 |
| 日活用户 | 500 | 2000 | 5000 |
| 订单数/日 | 1000 | 5000 | 15000 |
| 数据库大小 | 10 GB | 50 GB | 150 GB |

### 9.2 资源预估

#### 9.2.1 应用节点

```
需求副本数 = ceil((峰值 QPS × 单请求耗时) / (单 Pod 吞吐量 × 安全系数))
         = ceil((500 QPS × 0.2s) / (100 QPS × 0.7))
         = ceil(100 / 70)
         = 2 副本
```

#### 9.2.2 数据库

| 规模 | 内存 | CPU | 磁盘 | 备份 |
|------|------|-----|------|------|
| 小（< 100 GB） | 16 GB | 4 核 | 200 GB SSD | 1 TB |
| 中（100-500 GB） | 32 GB | 8 核 | 1 TB SSD | 5 TB |
| 大（500 GB+） | 64 GB | 16 核 | 2 TB NVMe | 20 TB |

### 9.3 性能拐点

| 拐点 | 现象 | 解决方案 |
|------|------|---------|
| QPS > 1000 | 应用 CPU 100% | 增加副本 / 水平扩展 |
| DB 连接 > 150 | 连接池溢出 | 增加连接池 + 优化慢查询 |
| 缓存命中率 < 50% | DB 压力上升 | 调整缓存策略 |
| 磁盘 > 80% | 写入变慢 | 归档历史数据 / 扩容 |
