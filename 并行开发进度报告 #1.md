## 三、技术亮点

### 1. 主要联系人自动切换
```rust
/// 取消所有主要联系人
async fn clear_primary_contacts(&self, supplier_id: i32) -> Result<(), AppError> {
    let contacts = supplier_contact::Entity::find()
        .filter(supplier_contact::Column::SupplierId.eq(supplier_id))
        .filter(supplier_contact::Column::IsPrimary.eq(true))
        .all(&*self.db)
        .await?;

    for contact in contacts {
        let mut contact_active: supplier_contact::ActiveModel = contact.into();
        contact_active.is_primary = Set(false);
        contact_active.update(&*self.db).await?;
    }

    Ok(())
}
```

**功能说明**:
- 当设置某个联系人为主联系人时，自动将该供应商的其他联系人设为非主要
- 使用事务保证数据一致性
- 避免多个联系人同时为主要联系人的数据冲突

### 2. 联系人排序优化
```rust
let contacts = supplier_contact::Entity::find()
    .filter(supplier_contact::Column::SupplierId.eq(supplier_id))
    .order_by_asc(supplier_contact::Column::IsPrimary)  // 主要联系人优先
    .order_by_asc(supplier_contact::Column::ContactName) // 按姓名排序
    .all(&*self.db)
    .await?;
```

**功能说明**:
- 主要联系人始终排在第一位
- 其他联系人按姓名拼音排序
- 方便用户快速找到主要联系人

## 四、遇到的问题与解决

### 问题 1：主要联系人唯一性约束
**问题**: 数据库层面无法直接约束一个供应商只能有一个主要联系人

**解决方案**:
- 应用层保证：在设置主要联系人前，先取消该供应商所有联系人的主要联系人状态
- 使用事务：保证操作的原子性
- 数据库约束：添加 UNIQUE 约束（supplier_id, is_primary），但 PostgreSQL 不支持部分唯一索引

**权衡**:
- 最终选择应用层保证 + 事务的方式
- 简单可靠，易于维护
