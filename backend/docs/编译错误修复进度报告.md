### 1. 重复 Entity 定义错误（高优先级）

**涉及文件**（10 个文件）:
- ⏳ `src/models/inventory_count_item.rs`
- ⏳ `src/models/inventory_adjustment_item.rs`
- ⏳ `src/models/inventory_reservation.rs`
- ⏳ `src/models/customer.rs`
- ⏳ `src/models/assist_accounting_dimension.rs`
- ⏳ `src/models/assist_accounting_record.rs`
- ⏳ `src/models/assist_accounting_summary.rs`
- ⏳ `src/models/business_trace_chain.rs`
- ⏳ `src/models/business_trace_snapshot.rs`
- ⏳ `src/models/business_trace_view.rs`
- ⏳ `src/models/purchase_order_item.rs`
- ⏳ `src/models/purchase_receipt_item.rs`

**问题描述**:
这些文件同时包含了两种 Entity 定义方式：
1. 手动定义的 `Entity` struct
2. `#[derive(DeriveEntityModel)]` 自动生成的 `Entity`

**错误示例**:
```
error[E0428]: the name `Entity` is defined multiple times
  --> src\models\inventory_count_item.rs:17:35
   |
8  | pub struct Entity;
   | ------------------ previous definition
...
17 | #[derive(Clone, Debug, PartialEq, DeriveEntityModel, Eq)]
   |                                   ^^^^^^^^^^^^^^^^^ redefined here
```

**修复方案**:
移除手动定义的 Entity struct 和 EntityName 实现，只保留 DeriveEntityModel。

**预计影响**: 每个文件约 10-15 行代码修改

### 2. 缺失 primary key 错误（高优先级）

**涉及文件**（5 个文件）:
- ⏳ `src/models/supplier.rs`
- ⏳ `src/models/supplier_contact.rs`
- ⏳ `src/models/supplier_category.rs`
- ⏳ `src/models/supplier_grade.rs`
- ⏳ `src/models/supplier_qualification.rs`
- ⏳ `src/models/supplier_blacklist.rs`

**问题描述**:
这些 Model 定义中缺少 `#[sea_orm(primary_key)]` 属性。

**错误示例**:
```
error: Entity must have a primary key column. See <https://github.com/SeaQL/sea-orm/issues/485> for details.
 --> src\models\supplier.rs:3:35
  |
3 | #[derive(Clone, Debug, PartialEq, DeriveEntityModel, Eq)]
  |                                   ^^^^^^^^^^^^^^^^^
```

**修复方案**:
在每个 Model 的第一个字段（通常是 `id`）上添加 `#[sea_orm(primary_key)]` 属性。

**预计影响**: 每个文件约 1-2 行代码修改

## 五、技术总结

### 学到的经验

1. **批量操作的风险**: PowerShell 批量替换容易导致文件编码问题，应使用 IDE 工具逐个精确修改
2. **SeaORM 版本差异**: 项目使用的 SeaORM 版本可能有特定的 API 要求，需要仔细检查文档
3. **类型系统严格性**: Rust 的类型系统非常严格，引用和所有权的区别很重要

### 最佳实践建议

1. **代码生成**: 考虑使用 SeaORM 的代码生成工具，避免手动定义 Entity
2. **增量修改**: 每次只修改少量文件，然后立即编译验证
3. **版本控制**: 每次成功编译后提交代码，便于回滚
