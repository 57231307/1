### 3. 性能测试脚本 ✅

**实施内容:**

创建了两个性能测试脚本:

#### 3.1 wrk 测试脚本 (`scripts/performance_test_wrk.sh`)

**支持的测试:**
- ✅ 健康检查测试
- ✅ 采购合同列表/详情测试
- ✅ 销售合同列表/详情测试
- ✅ 固定资产列表/详情测试
- ✅ 预算管理列表/详情测试
- ✅ 高并发测试 (500 连接)

**测试参数:**
```bash
connections=100    # 并发连接数
threads=8          # 线程数
duration=30s       # 测试持续时间
```

**输出指标:**
- Latency (延迟): Avg, Stdev, Max, 百分位数
- Req/Sec (吞吐量)
- 总请求数

#### 3.2 Apache Bench 测试脚本 (`scripts/performance_test_ab.sh`)

**支持的测试:**
- ✅ 健康检查测试
- ✅ 所有 GET 端点测试
- ✅ 高并发测试 (2000 请求，500 并发)

**测试参数:**
```bash
requests=1000      # 总请求数
concurrency=100    # 并发数
```

**输出指标:**
- Requests per second
- Time per request
- Transfer rate
- Connection Times

**使用方法:**
```bash
# 安装依赖
# Windows: choco install wrk
# macOS: brew install wrk

# 运行测试
cd scripts
chmod +x performance_test_wrk.sh
./performance_test_wrk.sh
```

## 📈 性能基准建议

### 推荐性能指标

**低负载场景** (100 并发):
- 响应时间：< 100ms
- 吞吐量：> 1000 req/s
- 错误率：< 0.1%

**中负载场景** (500 并发):
- 响应时间：< 200ms
- 吞吐量：> 3000 req/s
- 错误率：< 0.5%

**高负载场景** (1000 并发):
- 响应时间：< 500ms
- 吞吐量：> 5000 req/s
- 错误率：< 1%

### 性能优化建议

1. **数据库优化**:
   - 添加索引
   - 优化查询
   - 使用连接池

2. **缓存策略**:
   - Redis 缓存
   - 内存缓存
   - HTTP 缓存

3. **代码优化**:
   - 异步处理
   - 批量操作
   - 减少锁竞争

## 📚 最佳实践

### API 设计

1. **RESTful 原则**:
   - 使用名词复数作为资源名
   - 使用 HTTP 方法表示操作
   - 使用状态码表示结果

2. **版本控制**:
   - URL 中包含版本号 (`/api/v1/`)
   - 保持向后兼容
   - 明确标记废弃的 API

3. **错误处理**:
   - 统一的错误格式
   - 清晰的错误消息
   - 合适的状态码

## 🐛 故障排查

### 常见问题

**Q1: Swagger UI 无法访问**
```bash
# 检查服务是否运行
curl http://localhost:8080/api/health

# 检查端口占用
netstat -ano | findstr :8080
```

**Q2: Postman 返回 401**
- 确保已执行登录
- 检查 Token 是否过期
- 验证环境变量配置

**Q3: 性能测试失败**
- 检查服务状态
- 验证 JWT Token
- 确认测试工具安装

**Q4: 文档生成失败**
```bash
# 清理并重新生成
cargo clean
cargo doc --no-deps --open
```
